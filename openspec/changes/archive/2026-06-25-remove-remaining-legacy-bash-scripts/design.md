## Context

This design implements the proposal for `remove-remaining-legacy-bash-scripts` and follows the architectural constraints in `architecture.md` and finalized decisions in `decisions.md`. The baseline control-plane migration is complete, but behavior ownership still remains in shell for host networking, service startup, and parts of renderer/bootstrap execution. This design details how to remove those remaining shell-owned runtime paths while delivering a single externally breaking release that also removes wrapper command paths and `.env` compatibility.

Implementation spans multiple domains:
- Operator command surface and CLI compatibility removal
- Host-network topology/firewall/NAT mutation ownership
- Service container startup contracts and runtime-init execution
- Renderer completion for all documented service paths
- State persistence for startup contracts and MAC/interface allocations
- Migration/rollback bundle generation and strict release-gate updates

## Goals / Non-Goals

**Goals:**
- Make Go the only mutation authority for command, host-network, runtime bootstrap, and runtime-critical rendering behavior.
- Remove wrapper script command paths and `.env` profile compatibility in this release.
- Provide deterministic startup via versioned JSON startup contracts consumed by shared-library-based per-service runtime-init binaries.
- Preserve deterministic MAC/interface role behavior, but derive it from typed plan/catalog/state inputs.
- Persist startup contracts and allocation outcomes as deployment and operation artifacts for observability and rollback.
- Ship a single migration/rollback bundle contract generated by `vcpe` for upgrade safety.
- Ensure all documented services (`bng`, `gateway`, `routerd`, `webpa`, `xb10`, `client`) pass direct `vcpe` integration and representative smoke gates on new paths.

**Non-Goals:**
- Introduce Kubernetes/multi-node orchestration or change Podman-first architecture.
- Redesign user-facing deployment manifest semantics beyond what is needed for startup-contract/runtime ownership.
- Keep any permanent shell fallback for host-network mutation, renderer execution, or startup control flow.
- Maintain `.env` profile compatibility after this change ships.

## Decisions

1. Command-surface cut and grammar
- Decision: Remove top-level wrappers now and consolidate service workflows under `vcpe service <name> ...`.
- Rationale: This matches explicit breaking decisions and avoids carrying dead compatibility paths into runtime migration completion.
- Alternatives considered:
  - Keep wrappers one more release: rejected by explicit decision override.
  - Use top-level service nouns (`vcpe bng`): rejected for weaker command-tree consistency.

2. Host-network ownership migration
- Decision: Replace platform shell host-network scripts with Go Host Network Controller phases integrated in apply/reconcile.
- Rationale: Host topology/NAT/firewall are high-risk mutation boundaries and must be typed and testable.
- Alternatives considered:
  - Shell retained behind subprocess adapters: rejected as hidden second control-plane.
  - Defer host-network migration: rejected because it leaves core behavior shell-owned.

3. Runtime bootstrap implementation
- Decision: Add shared runtime-init Go library with thin per-service binaries in service images.
- Rationale: Startup requires ordered logic that compose alone cannot guarantee.
- Alternatives considered:
  - Compose-only bootstrap: rejected due to missing ordered interface/readiness/config guarantees.
  - Per-service independent bootstrap implementations: rejected due to drift risk and test duplication.

4. Startup contract model
- Decision: Runtime-init consumes versioned JSON startup contract rendered per service instance; contracts persisted as operation artifact and deployment-scoped state.
- Rationale: Single explicit typed startup input improves determinism, diagnostics, rollback, and schema evolution.
- Alternatives considered:
  - Infer from environment variables/files: rejected due to ambiguity and weaker forensic reproducibility.
  - Couple contract version to manifest version: rejected to allow independent internal evolution.

5. Renderer completion strategy
- Decision: Expand typed Go renderers for all documented services; fail fast before mutation for unsupported gaps.
- Rationale: Runtime rendering is part of deployment truth and cannot remain script-owned.
- Alternatives considered:
  - Permanent Python/bash adapter: rejected as indefinite split architecture.

6. Migration/rollback packaging
- Decision: `vcpe` generates a single versioned migration bundle per deployment/customer containing upgrade input plus rollback payload.
- Rationale: Combined breaking changes (wrappers + runtime bash + `.env`) require deterministic rollback package.
- Alternatives considered:
  - Manual migration docs only: rejected as operationally unsafe.
  - Separate upgrade and rollback artifacts: rejected for higher operator error risk.

7. Failure and rollback semantics
- Decision: Runtime-init failures are hard deployment failures with bounded rollback of current operation.
- Rationale: Partial convergence would hide broken startup contracts and leave invalid runtime states.
- Alternatives considered:
  - Allow partial convergence: rejected due to operational ambiguity.

8. Release gate strength
- Decision: Every documented service path must pass direct `vcpe` integration plus representative Podman smoke coverage.
- Rationale: Broad ownership transfer and compatibility removals require stronger evidence.
- Alternatives considered:
  - Reduced service-by-service gate: rejected as insufficient for one-shot breaking cut.

## Risks / Trade-offs

- [Risk] Combined breaking surface (command path + profile format + runtime ownership) increases upgrade complexity.
  - Mitigation: Require pre-upgrade migration bundle, add rollback bundle validation, and block apply without artifacts.
- [Risk] Runtime-init ordering regressions could cause hard-to-debug startup failures.
  - Mitigation: Shared bootstrap phase state machine, phase-specific tests, and first-class startup diagnostics in status/logs.
- [Risk] Host-network migration may regress bridge/NAT/firewall behavior across host types.
  - Mitigation: Capability preflight, typed plan diffs, Linux+macOS delegation test matrix, and guarded staged internal cutover flags.
- [Risk] Full-service renderer migration in one release can uncover late parity gaps.
  - Mitigation: Fail-fast unsupported checks before mutation, expand per-service integration fixtures early, gate by documented support surface.
- [Risk] Persisted startup contracts and allocations may drift from runtime if updates are not atomic.
  - Mitigation: Transactional operation artifact writes tied to reconcile phases and checksum validation at runtime-init start.
- [Trade-off] Faster architectural cleanup now vs. larger one-time customer migration burden.
  - Mitigation: Provide explicit migration tooling, docs, and rollback path in same release.

## Migration Plan

1. Internal staged cutover (implementation phase)
- Add feature flags for host-network controller, runtime-init contract path, and renderer completion path.
- Migrate service/component flows incrementally behind flags while preserving current release behavior by default.
- Build parity tests on both old/new internal paths until all documented services pass new-path gates.

2. Artifact and state readiness
- Introduce startup contract JSON schema/versioning and persistence paths.
- Add migration bundle generation (`vcpe migration bundle create`) and validation command paths.
- Persist allocation/startup artifacts for status/log/rollback integration.

3. External breaking release cut
- Flip default paths to Go-owned host-network/runtime/renderer.
- Remove wrapper scripts and `.env` compatibility translation paths.
- Enable `vcpe service <name> ...` as canonical service workflow surface.
- Enforce strict release gates and migration-bundle preconditions.

4. Rollback strategy
- If deployment fails during runtime-init/host-network apply, trigger bounded rollback using persisted rollback payload from migration bundle.
- Preserve per-operation diagnostics and startup contracts for forensic inspection.

5. Post-cut stabilization
- Remove now-obsolete internal feature flags after confidence window.
- Keep migration/rollback bundle functionality for upgrade safety across subsequent releases.

## Open Questions

- Which diagnostics fields should be considered stable API in `vcpe status` output vs. best-effort operational metadata?
